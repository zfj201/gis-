## 1. 项目背景与问题定义

### 1.1 背景
传统 WebGIS 系统通常要求用户理解图层、字段、空间关系和查询表达式。对于文旅场景中的普通用户，这些操作门槛高、学习成本大。用户更习惯自然语言问法，例如：
- `鼓楼区公园有多少个`
- `标准名称为南二环的道路街巷100米内的门牌号码`
- `面积小于100的宗地院落80米内的道路街巷`

### 1.2 核心问题
本项目要解决的本质问题是：
**如何将自然语言稳定、可解释地映射为可执行的空间查询，并把结果直观反馈到地图和对话中。**

可拆分为四个子问题：
1. 语义理解问题：用户表达不规范、歧义多、上下文弱。
2. 查询安全问题：不能让模型直接拼 SQL；必须字段合法、操作符可控。
3. 空间执行问题：点/线/面数据差异大，跨图层缓冲更复杂。
4. 可用性问题：模型不稳定时系统也必须可用，不能“全挂”。

### 1.3 目标用户与使用方式
- 用户类型：文旅管理人员、规划人员、演示人员、普通访客。
- 使用方式：左侧聊天输入自然语言，右侧地图查看高亮结果、缓冲区和图层状态。

## 2. 项目目标、边界与原则

### 2.1 项目目标
1. 实现自然语言到结构化 DSL 的转换。
2. 通过 ArcGIS REST API 执行空间查询。
3. 在地图上高亮点/线/面结果并支持缓冲区可视化。
4. 支持多 FeatureLayer 动态接入、显隐控制、服务级视角定位。
5. 模型优先、规则兜底，确保稳定可用。

### 2.2 当前边界
1. 单轮单目标图层查询，不做多目标图层并行聚合。
2. 主打检索与统计，不做复杂拓扑分析（如 union/clip/overlay 全族）。
3. 地名 geocode 可接入但非强内置，未配置时会追问。
4. 非空间对话走普通聊天模型，不触发 GIS 查询执行。

### 2.3 核心设计原则
1. 可执行优先：先保证“能查到”，再追求“更聪明”。
2. 可解释优先：每次都给出 DSL、查询计划、解析来源。
3. 安全优先：字段白名单、Schema 校验、运算符约束。
4. 韧性优先：模型失败自动回退规则，系统不断电。
5. 可扩展优先：图层动态注册，不写死单图层结构。

## 3. 设计理念与核心思想

### 3.1 设计理念
本项目不是“做一个聊天机器人”，而是“做一个**可控的空间查询编排器**”。  
LLM 在系统中不是最终执行者，而是“语义建议器”；真正执行前必须经过：
1. 结构化校验
2. 语义归一化
3. 图层字段合法性校验
4. 查询编译与空间执行器约束

### 3.2 核心思想：LLM + Rule + DSL + Compiler 四层闭环
1. **LLM 层**：提升自然语言理解上限，覆盖复杂问法。
2. **Rule 层**：稳定底线能力，保障故障时可用。
3. **DSL 层**：把“语言”转成“机器可控意图”。
4. **Compiler 层**：把 DSL 变成 ArcGIS `/query` 参数，且严格防越界。

这个四层闭环是本系统最关键的方法论：  
**语义自由输入 -> 结构化约束 -> 空间可执行 -> 结果可解释。**

### 3.3 为什么必须 DSL
DSL 的价值不是“多写一层”，而是解决三个核心矛盾：
1. 模型表达自由 vs 查询执行确定性。
2. 用户自然语言模糊 vs GIS 接口参数刚性。
3. 多数据源演进 vs 前后端契约稳定。

## 4. 系统总体架构

## 4.1 技术栈
1. 前端：Vue 3 + Vite + TypeScript + ArcGIS JS API
2. 后端：Node.js + Fastify + TypeScript
3. 共享契约：`@gis/shared`（DSL/DTO/Schema）
4. 外部服务：ArcGIS/GeoScene FeatureLayer REST，OpenAI-compatible LLM（Groq/OpenRouter）

## 4.2 架构分层
1. 展示层（Frontend）
2. 编排层（Fastify API）
3. 语义层（semantic + semantic-llm + normalizer）
4. 执行层（compiler + spatial-executor + arcgis）
5. 资源层（layer-registry + persistent catalog）

## 4.3 关键模块
1. `layer-registry`：图层注册、元数据发现、持久化。
2. `semantic.ts`：规则解析和兜底。
3. `semantic-llm.ts`：模型解析、重试、自修复、失败归因。
4. `semantic-normalizer.ts`：模型与规则输出统一归一化。
5. `compiler.ts`：DSL 编译为 QueryPlan。
6. `spatial-executor.ts`：分页执行、跨图层缓冲、几何合并。
7. `narrator.ts`：结果自然语言总结。
8. `MapViewPanel.vue`：图层渲染、命中高亮、缓冲区绘制、服务视角跳转、地图点击选中。

## 5. 数据模型与接口契约

## 5.1 DSL 核心结构（SpatialQueryDSL）
关键字段：
1. `intent`：`search | count | group_stat | nearest | buffer_search`
2. `targetLayer`：目标图层 `layerKey`
3. `attributeFilter[]`：字段条件（支持 `= like > >= < <=`）
4. `spatialFilter`：`buffer/intersects/nearest` 及半径、center、sourceLayer/sourceAttributeFilter
5. `aggregation`：`count/group_count`
6. `limit`：当前上限扩展到 `500000`（安全阈值同级）
7. `output`：字段与几何返回策略

## 5.2 QueryPlan
编译后可直接请求 ArcGIS `/query` 的参数对象，核心字段：
1. `where`
2. `geometry/geometryType/spatialRel/distance/units`
3. `outFields/returnGeometry`
4. `resultOffset/resultRecordCount`
5. `returnCountOnly` 或统计参数

## 5.3 API 核心接口
1. `POST /api/chat/query`：统一入口（推荐）
2. `POST /api/semantic/parse`：只做语义解析
3. `POST /api/spatial/execute`：只做执行
4. `GET /api/layers/catalog`：图层目录
5. `POST /api/layers/catalog/register`：注册服务
6. `PATCH /api/layers/catalog/layer/:layerKey`：显隐/可查询状态
7. `DELETE /api/layers/catalog/service/:serviceId`：删除服务
8. `GET /api/layers/meta`：图层元数据
9. `POST /api/layers/export/features-page`：分页导出查询

## 5.4 扩展响应元数据
当前增加了执行可观测字段（可选）：
1. `executionMeta.truncated`
2. `executionMeta.safetyCap`
3. `executionMeta.fetched`
4. `executionMeta.requestedLimit`

用途：在前端提示“已触发安全阈值，结果截断”。

## 6. 端到端流程（核心链路）

## 6.1 空间问题链路
1. 用户输入自然语言。
2. `/api/chat/query` 判断是否空间问题。
3. 空间问题进入 `parseQuestionSmart`：
   - 模型优先（Groq/OpenRouter）
   - 失败则规则回退
4. 得到 DSL 后进入 `executeDsl`：
   - 编译 QueryPlan
   - 分页执行 ArcGIS 查询
   - 返回 features 与 executionMeta
5. `narrator` 生成自然语言总结。
6. 前端显示 summary + DSL + queryPlan + 解析来源。
7. 地图高亮结果，若 queryPlan 含缓冲参数则绘制缓冲区。

## 6.2 非空间问题链路
1. 关键词规则判定为非空间问题。
2. 直接走 `answerGeneralQuestion`（普通聊天）。
3. 不触发 GIS 查询执行，不绘制高亮。

## 6.3 模型失败链路
1. 模型超时/401/非JSON/Schema不合规/一致性冲突。
2. 自动进入规则解析。
3. 响应包含：
   - `parserSource=rule_fallback`
   - `parserFailureReason`
   - `parserFailureDetail`
4. 用户仍能获得可执行结果，系统可用性不受致命影响。

## 7. 语义解析策略（模型与规则对齐）

## 7.1 模型侧约束
1. System Prompt 固定六段结构（角色、契约、DSL约束、意图映射、跨图层硬约束、动态图层上下文）。
2. few-shot 针对高频失败样例：
   - `标准名称为...`
   - `有多少个`
   - `OBJECTID：...`
   - `面积小于...`
   - `周长不超过...`
   - `你好`（非空间）
3. 模型请求启用结构化输出优先（provider 支持时）。
4. 首次失败会触发单次“自修复重试”。

## 7.2 规则侧策略
1. 显式条件优先：字段+操作符+值。
2. 运算词映射：
   - 为/等于/就是 -> `=`
   - 包含/相关 -> `like`
   - 小于/大于/不超过/至少 -> 数值比较符
3. 同义词映射：
   - `面积` -> `SHAPE__Area`
   - `周长/长度` -> `SHAPE__Length`
   - `objectid/编号/id` -> `objectid`

## 7.3 归一化层（关键）
`semantic-normalizer` 统一处理模型输出与规则输出，目标是“同句同义”：
1. 清理噪音尾词（如“有多少个”）。
2. 数值比较从 like 纠偏为 `> >= < <=`。
3. 字段名映射与合法性校验。
4. sourceLayer/sourceAttributeFilter 合法性校验。

## 8. 空间执行策略与算法要点

## 8.1 编译器策略（compiler）
1. 字段白名单校验。
2. 运算符-字段类型匹配校验。
3. 数值字段比较强制数字化。
4. `outFields` 自动补 `objectIdField`，保证选择/导出链路稳定。

## 8.2 分页执行策略（取消 2000 固定上限）
当前策略：
1. 对用户无感，不再暴露 2000 限制。
2. 后端自动分页拉取结果。
3. 采用 `QUERY_MAX_FEATURES=500000` 作为安全阈值。
4. 超阈值不报错，返回截断结果并标记 `executionMeta.truncated=true`。
5. 通过 `QUERY_PAGE_SIZE`、`QUERY_MAX_PAGES` 控制稳定性。

## 8.3 跨图层缓冲（线/面源）
1. 先按 `sourceLayer + sourceAttributeFilter` 查询源要素。
2. 多源时自动合并几何：
   - 线：paths 合并
   - 面：rings 合并
   - 点：points 合并
3. 将合并几何作为目标图层空间过滤条件执行。

## 8.4 缓冲区可视化触发规则
按 `queryPlan` 触发：只要存在 `geometry + geometryType + distance` 即绘制。

## 9. 前端交互与可解释性设计

## 9.1 聊天界面
每条回答可展示：
1. 自然语言总结
2. 命中要素数量
3. 目标图层
4. 解析来源（模型/规则回退）
5. 回退原因与回退细节
6. 结构化 DSL
7. 查询计划

## 9.2 图层管理（地图右侧）
1. 动态添加 FeatureServer。
2. 服务级折叠/展开。
3. 服务级显隐开关。
4. 服务级“移动视角”按钮（替代右键菜单）。
5. 图层级显隐开关。
6. 图层右键菜单导出 Shapefile。

## 9.3 选择与导出
1. 选中来源 = 地图点击选中 ∪ 当前查询高亮命中。
2. 导出策略：选中优先；无选中则整层分页导出。
3. 导出格式：Shapefile ZIP，含 `.cpg=UTF-8`。

## 9.4 大结果渲染策略
1. 保留全量高亮语义。
2. 高亮渲染采用分批 `addMany` + `requestAnimationFrame`，减轻主线程阻塞。

## 10. 项目创新点（可用于答辩重点）

## 10.1 创新点一：模型优先 + 规则兜底 + 归一化裁决三层语义架构
不是简单“模型失败就回退”，而是：
1. 模型先做高语义覆盖
2. 规则兜底保可用
3. 归一化层做语义对齐，减少同句不同结果

## 10.2 创新点二：动态图层目录驱动语义与执行
图层可运行时注册，语义解析不依赖写死字段；prompt 动态注入图层和字段元数据，实现“多图层可查询”。

## 10.3 创新点三：跨图层缓冲自动合并源几何
实现了“某线/某面多少米内的另一图层”自然问法，支持多源命中后自动合并执行，不再强制用户二次拆分。

## 10.4 创新点四：可解释性工程化落地
返回解析来源、回退原因、回退详情、DSL、QueryPlan，形成从输入到执行的可追踪闭环。

## 10.5 创新点五：查询上限从硬阈值转为“安全阈值 + 透明提示”
取消 2000 固定限制，改为可配置大规模分页 + 截断元信息，兼顾“用户体验”和“系统稳定”。

## 11. 核心优势总结

1. 稳定：模型异常不导致系统不可用。
2. 安全：强约束 DSL，字段和运算符严格校验。
3. 可扩展：多服务多图层动态接入。
4. 可解释：可视化调试信息完整，便于开发和答辩。
5. 可演示：聊天、地图、图层管理、导出闭环齐全。
6. 可工程化：配置驱动、模块化、可持续迭代。

---

## 12. 当前不足与技术债

1. 多图层并行聚合尚未实现（一次仅单目标图层）。
2. `nearest/intersects` 在 DSL 层可表达，但主执行路径仍以 `buffer/search/count/group` 为核心。
3. 超大结果时后端内存和前端渲染压力仍然较高（虽然已做分页与分批渲染）。
4. 非空间问题识别目前主要靠关键词规则，边界样例仍可能误判。
5. 地名 geocode 依赖外部服务，未内置时部分问句需要追问。
6. 导出当前主要是 Shapefile，格式生态还可扩展（GeoJSON/CSV/FileGDB）。
7. 评测体系仍偏手工，缺少自动化 benchmark 与持续监控看板。

## 13. 风险与治理策略

## 13.1 语义风险
风险：模型输出不稳定。  
治理：结构化输出约束 + Schema 校验 + 单次修复 + 规则回退。

## 13.2 查询风险
风险：非法字段/错误类型导致错误查询。  
治理：编译器字段白名单、字段类型校验、数值比较强约束。

## 13.3 性能风险
风险：大结果量导致接口慢或前端卡。  
治理：分页执行、最大页数保护、安全阈值截断、前端分批渲染。

## 13.4 外部依赖风险
风险：LLM provider 或图层服务不可用。  
治理：模型失败回退规则、服务错误友好提示、日志可追溯。

## 17. 结语
本项目已从“单图层查询 Demo”演进为“可运行、可扩展、可解释”的空间语义系统原型。  
它的价值不在于某个单点算法，而在于把自然语言理解、GIS 执行、安全约束、可解释反馈和工程稳定性整合为一个闭环架构。

对毕设而言，这个项目具备了“研究问题明确、技术方案完整、工程实现充分、可演示性强、可继续迭代”的关键条件。  
后续只要补齐评测体系与研究对比实验，就可以形成一套完整且有说服力的毕业设计成果。
